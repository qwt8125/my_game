# v0.2 详细开发计划

## 1. 版本定位

v0.2 的目标不是扩大技术架构，而是把 MVP 从“核心功能能跑”推进到“玩家有目标、有消耗、有持续收益、有阶段挑战”。

核心方向：

- 补齐 MVP 中任务查看和任务领取的验收缺口。
- 增加装备强化，让金币和材料有稳定消耗场景。
- 增加挂机收益，让碎片化玩法成立。
- 增加 BOSS 刷新和挑战，提供阶段性目标。
- 增加简易 GM 后台，支持基础运营和问题处理。
- 继续优化现有前端体验，保证新增功能可通过浏览器完成。

## 2. 当前缺口判断

当前不是只差任务领取一个功能。

任务领取是最明确的 MVP 验收缺口，因为 PRD 中写了“玩家可以查看任务并领取奖励”，而代码目前只有 `tasks.json` 和 `task_progress` 表，尚无任务接口和前端页面。

除此之外，v0.2 还应该补充这些能力：

- 成长目标：任务系统、BOSS 目标。
- 资源消耗：装备强化消耗金币或材料。
- 离线/半离线收益：挂机收益。
- 运营能力：简易 GM 后台。
- 体验闭环：新增功能都需要可视化入口、反馈提示和错误提示。

## 3. 优先级范围

### P0 必做

P0 是 v0.2 的闭环基础，建议全部完成后再认为 v0.2 可发布。

#### 3.1 任务系统

状态：已完成基础版。

目标：

- 玩家可以查看任务列表。
- 战斗、升级、装备等行为可以推进任务进度。
- 任务达成后可以领取奖励。
- 奖励发放必须走数据库事务，并写入金币、掉落或奖励流水。

后端建议：

- `GET /api/tasks`：返回任务配置、当前进度、状态、奖励。
- `POST /api/tasks/{taskId}/claim`：领取已完成任务奖励。
- 战斗胜利后推进 `kill_monster` 类型任务。
- 等级变化后推进 `level_reach` 类型任务。
- 首版任务状态建议沿用 `task_progress.status`：
  - `0` 未完成
  - `1` 可领取
  - `2` 已领取

前端建议：

- 左侧导航增加“任务”。
- 任务页展示任务名称、目标、当前进度、奖励、状态。
- 可领取任务显示明确按钮。
- 领取成功后刷新角色、背包和任务状态。

验收标准：

- 玩家能看到 `tasks.json` 中配置的任务。
- 击杀指定怪物后任务进度增加。
- 达到等级条件后任务状态变为可领取。
- 领取任务后经验、金币或物品正确到账。
- 已领取任务不能重复领取。

#### 3.2 装备强化

状态：已完成基础版。

目标：

- 装备可以消耗金币或材料进行强化。
- 强化后装备属性提升，并影响角色战力。
- 强化失败或上限规则首版可以保持简单。

后端建议：

- 在 `inventory_items.extra_json` 中记录强化等级。
- `POST /api/equipment/enhance`：请求参数包含 `inventoryItemId`。
- 强化成本可先按等级递增：
  - +1 消耗 20 金币
  - +2 消耗 50 金币
  - +3 消耗 100 金币
- v0.2 首版建议强化必成，避免先引入失败补偿和概率争议。

前端建议：

- 背包装备项展示强化等级。
- 装备页或背包页提供“强化”按钮。
- 强化后展示属性变化和金币变化。

验收标准：

- 未穿戴和已穿戴装备都能展示强化等级。
- 金币不足时不能强化。
- 强化后金币扣除、属性提升、战力刷新。
- 强化相关金币变化写入 `currency_logs`。

#### 3.3 挂机收益

状态：已完成基础版。

目标：

- 玩家可以选择地图或怪物进行挂机。
- 挂机根据时间产生经验、金币和少量掉落。
- 收益结算必须有上限，避免无限累积。

后端建议：

- 新增挂机状态表，例如 `idle_sessions`。
- `POST /api/idle/start`：开始挂机。
- `POST /api/idle/claim`：结算并领取挂机收益。
- `GET /api/idle/status`：查看当前挂机状态和预计收益。
- 首版收益上限建议 8 小时。

规则建议：

- 只有满足地图进入等级才能挂机。
- 挂机收益低于手动战斗收益。
- 挂机掉落概率低于手动战斗。

验收标准：

- 玩家能选择可进入地图开始挂机。
- 间隔一段时间后可以领取经验和金币。
- 超过上限时间只按上限结算。
- 领取收益后角色经验、金币、背包正确变化。

### P1 应做

P1 是增强可玩性和运营能力，建议在 P0 稳定后继续。

#### 3.4 BOSS 刷新

状态：已完成基础版。

目标：

- 地图中有 BOSS 或精英怪。
- BOSS 有刷新状态和冷却时间。
- 击败 BOSS 可获得更高奖励。

后端建议：

- 新增 `boss_states` 表记录 BOSS 刷新状态。
- `GET /api/bosses`：查看 BOSS 状态。
- `POST /api/bosses/{bossId}/fight`：挑战 BOSS。
- 首版可以使用 SQLite 记录刷新时间，不引入 Redis。

验收标准：

- 玩家能看到 BOSS 是否可挑战。
- BOSS 未刷新时不能挑战。
- 击败后进入冷却。
- BOSS 奖励高于普通怪，并写入战斗与掉落日志。

#### 3.5 简易 GM 后台

状态：已完成基础版。

目标：

- 管理员可以查看账号和角色。
- 管理员可以发放金币或物品。
- 管理员可以重载配置或查看基础运行状态。

后端建议：

- 首版用配置文件维护管理员账号白名单。
- 增加 `/api/admin/**` 接口分组。
- GM 发放必须写入独立日志或复用流水 `reason=gm_grant`。

前端建议：

- 可先做极简后台页面，不追求完整管理系统。
- 入口可以隐藏，直接访问管理路径。

验收标准：

- 非管理员不能访问 GM 接口。
- 管理员能查询角色。
- 管理员能给指定角色发放金币或物品。
- GM 操作有日志可追踪。

### P2 可选

P2 不建议阻塞 v0.2 发布。

- 邮件系统：可放到 v0.3，除非 GM 发放需要站内信承接。
- Redis 接入：继续放到 v0.3，当前 SQLite 足够支持单服 MVP。
- 完整运营后台：放到后续版本，v0.2 只做最小 GM 能力。
- 排行榜性能优化：数据量小阶段不急。

## 4. 推荐开发顺序

1. 任务系统：补齐验收缺口，最适合作为 v0.2 第一项。
2. 装备强化：补金币和材料消耗，让刷装有后续追求。
3. 挂机收益：补碎片化玩法，但需要谨慎处理时间结算。
4. BOSS 刷新：补阶段目标和高价值掉落。
5. 简易 GM 后台：补运营处理能力。
6. v0.2 收尾体验优化：检查所有新增页面、错误提示、移动端布局和文档记录。

## 5. v0.2 发布验收标准

- 任务系统可完整查看、推进和领取奖励。
- 装备强化可消耗资源并提升战力。
- 挂机收益可开始、查看、领取，并有时间上限。
- BOSS 可查看刷新状态、挑战并进入冷却。
- 简易 GM 后台具备最小查询和发放能力。
- 所有新增玩家资产变化都在事务中完成。
- 金币、掉落、装备、任务奖励、GM 发放都有流水或日志。
- 前端可以完成 v0.2 所有核心流程。
- `mvn package -DskipTests` 或等效构建通过。
- PRD、开发清单和优化记录完成同步。

## 6. v0.2 不做范围

- 不接入 Redis。
- 不做完整邮件系统。
- 不做拍卖行、公会、PK、跨服。
- 不做复杂强化失败、保护符、降级、保底系统。
- 不做完整商业化充值。
- 不做复杂 BOSS 争夺和多人实时战斗。

## 7. 风险点

- 挂机收益容易造成资源膨胀，必须设置时间上限和较低收益率。
- 装备强化如果过早引入失败概率，会增加规则复杂度。
- GM 后台涉及权限，必须先做访问保护。
- 任务奖励、挂机奖励、BOSS 奖励都涉及资产变化，必须使用事务。
- 前端仍是原生 JS，功能增加后要控制页面复杂度，避免单文件过度膨胀。
