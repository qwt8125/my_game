# v0.3 战斗体验与在线状态计划

## 1. 阶段定位

v0.3 的主线不是单独接入 Redis，而是先把战斗体验从“一次请求结算整场”升级为“在线回合逐步返回，离线挂机后台结算”。

当前 v0.2 已完成任务、装备强化、挂机、BOSS 和简易 GM。下一步最影响玩家体感的是战斗过程：玩家在线时需要看到一轮一轮发生的攻击，离线或挂机时则不需要真实播放每个回合，只做后台时间结算。

当前状态：v0.3 P0/P1/P2 基础能力已完成；战斗会话 Redis Store、在线状态、BOSS 冷却缓存、排行榜 Redis 缓存、邮件/补偿进阶版和 SSE 战斗推送均已完成。

## 2. 核心目标

- 增加 `attackSpeed` 攻击速度属性，让角色、怪物和装备具备出手节奏差异。
- 新增在线战斗会话，支持启动战斗、推进下一次行动、查询当前状态。
- 前端战斗页从一次性结果展示改为实时回合日志展示。
- 离线挂机继续采用后台时间差结算，不模拟每一回合，避免性能浪费。
- 保留旧 `POST /api/battles/fight` 作为兼容接口，等新战斗页稳定后再考虑下线。

## 6. 攻击速度规则

攻击速度越高，出手间隔越短。

建议首版公式：

```text
actionIntervalMs = max(700, 2200 - attackSpeed * 12)
```

示例：

- `attackSpeed = 80`，出手间隔约 `1240ms`。
- `attackSpeed = 100`，出手间隔约 `1000ms`。
- `attackSpeed = 130`，出手间隔约 `700ms`，达到首版下限。

在线战斗第一版不必做复杂时间轴，可以按“下一次行动方 + 建议等待时间”推进；后续再扩展为更精细的行动条或技能冷却。

## 7. 在线战斗流程

```text
玩家选择地图和怪物
  -> POST /api/battles/start
  -> 后端创建 battle session，返回 battleId 和初始状态
  -> 前端显示战斗面板
  -> 前端优先连接 GET /api/battles/{battleId}/stream
  -> 后端按 suggestedDelayMs 自动推进行动并推送 state/action/finished 事件
  -> SSE 不可用时前端回退为按 suggestedDelayMs 调用 POST /api/battles/{battleId}/next
  -> 战斗结束后后端一次性结算奖励并写日志
```

关键原则：

- 每次 `next` 只推进一次行动，不一次性返回整场；SSE 流内部也复用该推进规则。
- 奖励只在战斗结束时结算一次。
- 进行中的战斗状态默认保存到 Redis 15 号库，SQLite 实现保留为回退。
- 旧接口暂时保留，用于兼容和回归测试。

## 8. 离线挂机流程

挂机不走在线回合会话。

```text
玩家开始挂机
  -> 后端记录 mapId、monsterId、startedAt
  -> 玩家离线或切页面
  -> 领取时按时间差计算经验、金币和少量掉落
  -> 收益封顶 8 小时
```

关键原则：

- 不逐回合模拟，避免后台消耗过高。
- 挂机收益低于在线战斗。
- 可参考攻击速度和战力影响单位时间收益，但要设置收益上限。
- 所有收益领取仍必须走数据库事务和流水。

## 9. 推荐开发顺序

1. 先做数据和配置：角色、怪物、装备增加 `attackSpeed`，战力公式同步。
2. 再做纯后端战斗会话：start、next、status 三个接口和测试。
3. 接着改前端战斗页：实时日志、HP 展示、自动推进。
4. 然后接入 BOSS 会话式战斗。
5. 最后评估是否需要 Redis、邮件和排行榜缓存。

当前推荐下一步：

本计划已完成，历史下一步中的世界 BOSS 已在 v0.4 落地。最新后续方向以 `doc/PRD.md` 的“当前状态”和“后续版本方向”为准。

1. 排行榜进阶：当玩家量明显增加后，将短 TTL 缓存升级为 Sorted Set 或定时榜单。
2. 更大玩法扩展：公会、拍卖行和运营活动配置。

## 10. 验收标准

- [x] 角色详情能看到攻击速度。
- [x] 装备或升级能影响攻击速度和战力。
- [x] 普通在线战斗不再一次性返回完整结果，而是每次返回一条行动。
- [x] 战斗结束前不会发放经验、金币和掉落。
- [x] 战斗结束后奖励、掉落、任务进度、日志与旧战斗结算一致。
- [x] 刷新页面后仍能恢复进行中的战斗状态。
- [x] 挂机收益仍可离线领取，不依赖前端在线轮询。

## 11. 风险点

- 战斗会话如果只放内存，服务重启会丢状态；首版建议至少落 SQLite。
- `next` 接口需要防重复点击，避免同一行动被推进两次。
- 攻击速度影响战力后，排行榜和已有角色数据需要迁移或重算。
- BOSS 冷却和战斗会话要避免重复结算。
- 前端自动轮询要在战斗结束、退出登录或切换怪物时停止。
