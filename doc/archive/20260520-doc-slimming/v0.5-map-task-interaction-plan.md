# v0.5 地图、NPC、任务与可交互地图方案

## 1. 背景与现状

当前 v0.4 世界 BOSS 基础版已完成，下一步可以回到玩家日常体验主线：地图探索、NPC 交互、任务密度和遇怪战斗。

现有实现状态：

- 地图配置在 `game/src/main/resources/config/maps.json`，当前只有 3 张地图：新手村外、毒蛇山谷、骷髅洞。
- 怪物配置在 `monsters.json`，当前只有 4 个普通怪物。
- 任务配置在 `tasks.json`，当前只有 2 个任务。
- 任务类型当前支持 `kill_monster` 和 `level_reach`。
- 地图接口当前是 `GET /api/maps` 和 `GET /api/maps/{mapId}`，地图详情返回怪物列表。
- 前端地图页当前是“查看怪物 -> 点击攻击”，战斗仍复用 `/api/battles/start` 和 SSE 行动推送。
- 当前没有 NPC 配置、NPC 接口、NPC 对话、接取任务、地图事件点、随机遇怪或地图格子状态。

## 2. 目标

- 增加地图数量、怪物数量、NPC 数量和任务数量，让玩家从新手阶段到中期有更连续的成长路径。
- 让地图不只是怪物列表，而是具备可点击、可交互、可触发事件的探索界面。
- 保持现有会话式战斗、SSE、任务奖励、邮件和掉落系统可复用。
- 第一版不做复杂实时多人同屏地图，避免把玩法扩展变成大规模前端引擎重写。

## 3. 方案一：内容扩容优先

### 做法

- 继续沿用现有地图列表和怪物列表 UI。
- 扩充 `maps.json`、`monsters.json`、`drops.json`、`tasks.json`。
- 新增 5 到 8 张地图，按等级段分布，例如：
  - 1-3 级：新手村、村外林地、废弃矿洞入口。
  - 4-7 级：毒蛇山谷、骷髅洞一层、矿洞深处。
  - 8-12 级：尸王殿外围、沃玛寺庙、赤月峡谷入口。
- 扩充任务为 20 到 30 个，仍以击杀、等级、装备强化、挑战 BOSS 等目标为主。

### 优点

- 实现最快，主要是配置扩展。
- 风险低，不影响现有战斗和任务闭环。
- 很适合先补内容空洞。

### 缺点

- 地图交互仍然偏静态，体验变化有限。
- NPC 和探索感不足，玩家仍像在点菜单。

## 4. 方案二：轻量可交互地图，推荐

### 做法

在方案一的内容扩容基础上，引入“地图点位”而不是完整格子地图。

新增配置：

- `npcs.json`：NPC ID、名称、所在地图、描述、可对话内容、可关联任务。
- `map-events.json`：地图点位 ID、所在地图、名称、类型、坐标、触发条件和触发动作。

地图点位类型建议：

- `npc`：点击后显示 NPC 对话，可接任务、交任务或查看提示。
- `monster_area`：点击后触发指定怪物战斗，复用现有 `/api/battles/start`。
- `random_encounter`：点击“探索”后按概率遇怪，也可能获得少量金币、材料或空事件。
- `portal`：地图跳转点，满足等级或任务条件后进入下一张地图。
- `boss_lair`：地图内 BOSS 入口，复用普通 BOSS 或世界 BOSS 展示。

后端新增接口：

- `GET /api/maps/{mapId}/scene`：返回地图基础信息、NPC、事件点、可用状态。
- `POST /api/map-events/{eventId}/trigger`：触发事件，返回对话、战斗启动结果、奖励或跳转信息。
- `POST /api/npcs/{npcId}/talk`：获取 NPC 对话和可执行动作。

任务系统扩展：

- 新增 `talk_npc`：与指定 NPC 对话。
- 新增 `explore_event`：触发指定地图事件。
- 新增 `clear_map`：完成某地图关键事件或击败指定数量怪物。
- 新增 `preTaskIds`：支持任务前置链。

前端调整：

- 地图页改为左侧地图列表，中间地图场景，右侧战斗/事件日志。
- 场景用 CSS 网格或绝对定位点位实现，不上复杂 Canvas。
- 点位可显示 NPC、怪区、传送门、宝箱、BOSS 等图标或短标签。
- 点击怪区或随机探索后，仍进入现有会话式战斗和 SSE 日志。

### 优点

- 能明显提升“在地图里行动”的感觉。
- 实现复杂度可控，不需要完整地图寻路和实时位置同步。
- 可以复用现有战斗、任务、掉落、BOSS、邮件奖励能力。
- 很适合文字传奇页游，既有探索感，又不会变成重客户端。

### 缺点

- 需要新增 NPC、事件点和任务前置关系的配置/校验。
- 任务系统要从“自动推进”扩展到“接取/交付/前置”的半 RPG 结构。

## 5. 方案三：格子探索地图

### 做法

- 每张地图配置为二维网格，例如 8x6 或 12x8。
- 玩家有当前地图坐标，可上下左右移动。
- 每个格子可能包含怪物、NPC、宝箱、传送门或随机事件。
- 后端记录角色当前位置和已探索格子。
- 移动时有概率触发遇怪战斗。

### 优点

- 探索感最强。
- 后续可以扩展迷宫、隐藏房间、采集点、区域控制等玩法。

### 缺点

- 改动最大，需要角色地图坐标、探索状态、移动接口、事件状态和前端网格 UI。
- 容易把当前轻量文字页游变成另一个复杂方向。
- 如果地图内容量不足，格子地图会显得空。

## 6. 推荐路线

推荐采用“方案二为主，方案一作为内容底座，方案三后置”的路线。实现策略明确为：**大地图 + 点位移动 + 像素角色模型**。

第一版不是纯静态点选，也不直接做完整自由移动。地图可以做成大于视口的黑白/灰阶场景图，玩家在地图中的位置先绑定到点位或轻量坐标；点击 NPC、怪区、宝箱、采集点、传送门后，前端播放人物移动到目标点位的动画，再触发对话、战斗、奖励、采集或跳转。

这样后续优化比较顺：

- 第一版可只记录 `currentMapId + currentNodeId`，表现为点位移动。
- 第二版可给点位增加 `x/y` 坐标和连线，支持人物沿路径移动。
- 第三版可支持大地图拖拽、缩放、小地图和更大的交互面积。
- 第四版如果玩法需要，再升级为格子或自由坐标探索。
- 任务、NPC、怪物、奖励都先绑定地图事件点，避免后续重做任务系统。

### 6.1 地图交互面积建议

- v0.5 第一版：单张地图逻辑尺寸建议为 `1600x1000` 或 `1800x1200`，前端可视区域约 `700x500`，通过点位跳转或轻量拖拽查看局部。
- v0.5 稳定后：扩展到 `2400x1600`，每张地图放置 20 到 40 个事件点。
- 不建议第一版做无限大地图；更适合用“新手村外、矿洞入口、矿洞深处、骷髅洞一层”等分区地图承接内容。

### 6.2 角色与像素模型建议

- 玩家角色第一版使用 48x64 或 64x64 像素模型。
- 至少准备 4 个方向：正面、背面、左侧、右侧。
- 动作优先级：待机、行走、攻击。
- 后续可按装备或职业替换武器、衣服颜色和轮廓。
- NPC、怪物、宝箱、草药、矿石、传送门统一使用像素风图标，尺寸建议 48x48 到 96x96。

### 6.3 数据结构预留

建议第一版配置就预留这些字段，避免后续推倒：

```json
{
  "id": "event_novice_wolf_area",
  "mapId": "map_novice_field",
  "name": "野兽出没区",
  "type": "monster_area",
  "x": 860,
  "y": 420,
  "sprite": "wolf_area",
  "targetMonsterIds": ["monster_wolf"],
  "requiredTaskIds": [],
  "nextEventIds": ["event_novice_portal"],
  "repeatable": true,
  "cooldownSeconds": 0
}
```

玩家状态第一版可以只保存：

- `character_id`
- `current_map_id`
- `current_node_id`
- `last_x`
- `last_y`

其中 `last_x/last_y` 可以先只供前端表现使用，后续再升级为真实坐标移动。

### v0.5 P0：内容扩容与任务链基础

- 新增地图到 8 到 10 张。
- 新增普通怪物到 15 到 20 个。
- 新增任务到 20 到 30 个。
- 任务按等级和地图拆成新手、矿洞、毒蛇、骷髅、寺庙等阶段。
- 任务类型先扩展到 `talk_npc`、`kill_monster`、`level_reach`、`explore_event`。
- 任务配置支持 `preTaskIds`，让任务按链路逐步解锁。

### v0.5 P1：NPC 与地图点位

- 新增 NPC 配置和 NPC 对话接口。
- 新增地图点位配置和地图场景接口。
- 每张地图配置 3 到 6 个可交互点位。
- 地图点位配置预留 `x/y` 坐标、sprite、前置任务、相邻点位和冷却字段。
- NPC 可展示对话、任务入口、地图提示。
- 怪区点位可直接发起战斗。
- 探索点位可触发随机遇怪或一次性奖励。

### v0.5 P2：前端交互地图

- 地图页改为黑白/灰阶大地图场景视图。
- 玩家角色使用像素模型展示在地图上。
- 点击点位时，先播放角色向目标点位移动的轻量动画，再触发事件。
- 点位展示状态：未解锁、可交互、已完成、冷却中。
- 事件日志与战斗日志整合。
- 战斗结束后刷新地图场景、任务状态和角色信息。
- 普通地图列表和怪物列表保留为兼容入口或调试入口。

### v0.5 P3：事件状态与可重复玩法

- 增加角色维度的地图事件状态表。
- 支持一次性宝箱、每日探索点、短冷却精英怪。
- 支持任务要求“完成某事件”或“探索某区域”。
- 后续再评估格子探索地图。

## 7. 建议优先实现的最小闭环

第一轮不建议直接做格子地图。建议先做以下闭环：

1. 新增 `npcs.json` 和 `map-events.json`。
2. 扩展地图详情为场景接口，返回 NPC、事件点、点位坐标和像素资源标识。
3. 新增 `talk_npc` 和 `explore_event` 两类任务。
4. 新增 2 张地图、3 个 NPC、6 个事件点、8 个任务作为样板。
5. 前端地图页展示黑白/灰阶地图、玩家像素模型和可点击点位。
6. 点击怪区或随机遇怪后复用现有战斗/SSE。
7. 点击 NPC 推进对话任务。
8. 战斗或事件结束后刷新任务和地图状态。

## 8. 验收标准

- 地图数量至少达到 8 张，普通怪物至少达到 15 个。
- 任务数量至少达到 20 个，并覆盖击杀、等级、NPC 对话、地图探索。
- 每张新地图至少有 1 个 NPC 或 2 个地图事件点。
- 玩家可在地图页点击 NPC、怪区、探索点或传送点，并看到角色移动到目标点位的表现。
- 地图视觉为黑白/灰阶底图，玩家、NPC、怪物、奖励和交互物使用像素风图片或精灵。
- 点击怪区或随机遇怪后，仍能通过现有 SSE 战斗日志完成战斗。
- NPC 对话或地图事件可推进任务。
- 前置任务未完成时，后续任务不展示或显示锁定状态。
- 普通 `/api/maps`、`/api/battles/start`、任务领取和普通 BOSS 流程不回归。

## 9. 不做范围

- 不做多人实时同屏地图。
- 不做复杂寻路和碰撞。
- 第一版不做完整自由移动；角色移动以点位移动或轻量坐标动画为主。
- 不做完整剧情编辑器。
- 不把地图状态放入 Redis 作为事实来源；玩家事件状态仍以 SQLite 为准。
- 不改变现有战斗结算、掉落和任务奖励事务原则。

## 10. 下一步建议

本计划已完成，v0.5.1-v0.5.11 已继续补充事件奖励、每日探索、状态管理、状态清理、玩家界面信息收口、地图交互打磨、点位聚合、移动端日志抽屉、地图缩放、拖拽视角、地图任务追踪、任务目标点位提示和任务目标定位元数据。最新后续方向以 `doc/PRD.md` 的“当前状态”和“后续版本方向”为准。

建议先进入 v0.5 P0/P1 的设计与实现：

- 先定义 NPC 和地图事件配置结构。
- 再扩展任务类型和任务前置。
- 然后实现地图场景接口。
- 最后改前端地图页，让玩家能在黑白/灰阶地图内操作像素角色，点击 NPC、怪区和探索点并触发轻量移动动画。
